# v0.6.3 — Watermark Pre-pass Speedup + Stage Progress UI

**Tanggal:** 2026-05-06
**Versi:** v0.6.2 → v0.6.3
**Tipe:** Patch (performance + UX)

## Latar Belakang

User melaporkan bahwa proses clone terasa **jauh lebih lambat** sejak v0.6.0 saat memproses banyak source Shopee (mis. 100 video) dengan auto-watermark removal aktif. UI hanya menampilkan "Processing clone #0... 0%" tanpa indikasi bahwa pre-pass watermark sedang berjalan, sehingga aplikasi terlihat hang.

## Root Cause

`VideoCloner._preprocess_watermark_all()` di v0.6.0 melakukan re-encode FFmpeg untuk setiap source yang punya watermark region — sebelum cloning dimulai. Ini punya 3 masalah:

1. **Encoder preset terlalu lambat.** Pre-pass pakai `libx264 -preset fast -crf 20`. Untuk source 480×854 / 52 dt, satu pre-pass makan ~26 dt. Kalau ada 100 source → 100 × 26 = ~43 menit pre-pass sebelum clone #0 mulai.
2. **Tidak paralel.** Semua pre-pass dijalankan berurutan, padahal CPU desktop modern punya 4–8+ core.
3. **Tidak ada feedback UI.** `progress.status='running'` saat pre-pass jalan, tapi `current_index=0` dan `percent=0`, jadi label muncul "Processing clone #0... 0%" terus selama pre-pass berlangsung.

## Perubahan

### `src/cloner.py`

#### 1. Encoder pre-pass diturunkan ke ultrafast
```diff
- '-c:v', 'libx264', '-preset', 'fast', '-crf', '20',
+ '-c:v', 'libx264', '-preset', 'ultrafast', '-crf', '23',
```
Output pre-pass adalah intermediate file yang langsung di-re-encode lagi oleh clone step (libx264 -crf 18 untuk method `standard`), jadi kualitas pre-pass tidak terlihat di output akhir. Aman untuk diturunkan.

**Benchmark (480×854, 52 dt, di sandbox 2-core):**
- `-preset fast -crf 20`: 26.4 dt
- `-preset ultrafast -crf 23`: 6.7 dt → **~3.9× lebih cepat**

#### 2. Pre-pass paralel dengan ThreadPoolExecutor
```python
cpu = os.cpu_count() or 2
workers = max(1, min(3, cpu // 2))
```
- 2-core CPU → 1 worker (sequential)
- 4-core CPU → 2 workers paralel
- 6+ core CPU → 3 workers paralel (cap)

Conservative cap mencegah oversubscription karena setiap ffmpeg call sudah multi-thread internal. Cancel juga di-extend untuk membunuh semua ffmpeg pre-pass yang sedang berjalan (bukan cuma satu).

#### 3. Stage progress reporting
```python
self.progress = {
    ...
    'stage': 'idle' | 'preprocessing' | 'cloning',
    'preprocess_index': int,
    'preprocess_total': int,
    'preprocess_current': str,
    ...
}
```
Field-field ini di-update saat pre-pass berlangsung, dipakai oleh UI untuk render label dan progress bar yang akurat.

#### 4. `{title}` tidak lagi pakai temp filename
Sebelumnya: setelah pre-pass, `self.source` di-set ke `wmrem_a1b2c3d4.mp4` (temp), dan `_resolve_filename()` ekstrak `{title}` dari sana. Akibat: file output bernama `wmrem_a1b2c3d4_clone01_2026-05-06.mp4`.

Sekarang: ada mapping `_source_titles[clean_path] = original_title`, dan `_resolve_filename()` pakai mapping itu. Output kembali ke nama source asli (mis. `shopee_rekomendasi tas anak bulu karakter_clone01_2026-05-06.mp4`).

### `src/web/app.js`

`updateProgress()` baca `data.stage`, `data.preprocess_index`, `data.preprocess_total`:

- **Stage = preprocessing**:
  - Progress bar = `(pp_index / pp_total) * 100%`
  - Counter = `pp_index/pp_total` (mis. `12/100`)
  - Label = `"Preprocessing watermark... 12/100"`
  - Subtext = `"Hiding watermark on 100 sources before cloning starts"`
- **Stage = cloning**: behavior lama tetap (mengikuti `current_index`, `total`).

## Verifikasi

Smoke test end-to-end (4 source Shopee, count=1 each, auto-Shopee enabled):

| Tahap | Old (v0.6.2) — estimasi | New (v0.6.3) — actual |
|-------|--------------------------|-----------------------|
| Pre-pass 4× | ~104 dt | ~21 dt |
| Clone 4× | ~6 dt | ~6 dt |
| **Total** | **~110 dt** | **~27 dt** |

Output filenames sekarang `src1_clone01_2026-05-06.mp4` (bukan `wmrem_xxxxxxxx_clone01_2026-05-06.mp4`).

Stage transitions di progress dict:
```
idle → preprocessing(0/4) → preprocessing(1/4) → … → preprocessing(4/4) → cloning(1/4) → … → completed
```

## Estimasi Improvement untuk User

User punya 100 source Shopee:

| Versi | Pre-pass (estimasi) |
|-------|---------------------|
| v0.6.2 | ~43 menit (26.4 dt × 100, sequential) |
| v0.6.3 di 2-core | ~11 menit (6.7 dt × 100, sequential) |
| v0.6.3 di 4-core | ~5–6 menit (6.7 dt × 100, 2 worker paralel) |
| v0.6.3 di 8-core | ~4 menit (6.7 dt × 100, 3 worker paralel) |

Plus UI sekarang menampilkan progress yang jelas, jadi tidak terkesan hang.

## Cara Pakai

Tidak ada perubahan workflow dari sisi user — semua perubahan transparent. Pre-pass otomatis lebih cepat dan paralel, UI otomatis menampilkan tahap pre-pass.

Kalau user mau full speed (skip pre-pass sama sekali), tinggal nonaktifkan toggle "Hide Watermark Region" di panel — pre-pass tidak akan jalan dan source asli (dengan watermark) langsung masuk ke clone step.
